# R2 汇报报告：AirfRANS 默认主路径的兼容式重训练与全流程验证

## 1. 工作目标

本轮工作的目标不是推翻现有二维代理模型，也不是直接把系统改造成三维工业级 CFD 平台，而是在保持当前 AirfRANS 主训练链路可运行的前提下，完成一次兼容式架构升级后的正式重训练，并验证以下三件事：

1. 默认 AirfRANS 训练、评测和推理链路仍然能够完整跑通。
2. 兼容式几何输入改造不会破坏现有 DeepONet 主干与固定维度 `branch_inputs` 契约。
3. 新增的通用二维翼型输入接口已经可以参与推理演示，但不会强制替代当前默认的 AirfRANS 主路径。
4. 后续阶段 7 中新增的 future native geometry backbone 仅为接口预留与 metadata 自描述能力，不代表系统已经完成原生可变长几何主干升级。

因此，这一轮的核心关注点是“工程链路是否稳定、分析输出是否完整、当前效果处于什么水平、下一步应如何继续优化”，而不是单纯追求某一个单项指标的最优值。

## 2. 方法原理

### 2.1 代理建模思想

当前系统采用的是一种神经算子代理思路。与传统的单点回归不同，模型学习的不是“某个固定输入对应一个固定输出标量”，而是学习一个更一般的映射关系：

- 输入：几何信息 + 工况信息 + 查询点位置
- 输出：该位置处的流场物理量，以及与整机气动性能相关的全局标量

从物理意义上看，这相当于让模型近似一个“条件流场响应算子”。当几何和来流条件确定后，模型应能够回答两个层面的预测问题：

1. 给定任意场点，输出局部流场变量；
2. 给定整段翼型或整组采样点，输出全局气动系数与分析型结果。

### 2.2 DeepONet 主结构

当前主模型仍然是 DeepONet 主线，没有被本轮重构推翻。其基本结构可以概括为：

- Branch 网络：编码几何与工况信息；
- Trunk 网络：编码查询点位置；
- 融合层：将 Branch latent 与 Trunk latent 组合；
- 多任务输出头：分别生成场变量、标量、特征等输出。

这种结构的优点是：

- 保持了任意点查询能力；
- 不依赖规则网格；
- 与 AirfRANS 这类基于非规则采样点的二维 RANS 数据更匹配。

### 2.3 为什么仍保留 fixed-dim branch_inputs

本轮没有直接把几何编码彻底改写成可变长原生主干，而是保留了当前固定维度 `branch_inputs` 契约，并把它显式定义成“兼容层表示”。这样做有三个原因：

1. 现有 checkpoint 和 AirfRANS `.npz` 数据已经固化了分支输入维度；
2. 当前 DeepONet 与 Geo-FNO 的主线都围绕这个输入契约组织；
3. 本轮目标是低风险重构，而不是大爆炸式替换主干。

因此，本轮把几何输入分成两条兼容通道：

- `legacy_fixed_features`
  - 默认模式；
  - 继续服务现有 AirfRANS 主路径；
- `encoded_geometry`
  - 对更原始的几何表示做轻量编码；
  - 最终仍映射为与现有 branch net 兼容的固定维度表示。

这意味着当前系统已经完成了“几何输入接口升级”，但还没有把主干彻底改造成原生可变长几何编码器。

### 2.4 阶段 7 之后如何理解 geometry backbone

后续阶段 7 又进一步补了 future native geometry backbone 的接口与 contract metadata，但这部分必须严格按“预留能力”理解，而不是按“已完成升级”理解。

当前应当把几何条件化分成两层：

- 当前稳定主线：
  - `geometry -> BranchInputAdapter -> fixed branch-compatible vector -> backbone`
- 未来预留主线：
  - `geometry -> native geometry backbone -> geometry latent / tokens -> future conditioning`

也就是说，当前仓库已经有了 future native geometry backbone 的统一接口骨架，但它的职责主要是：

1. 让 checkpoint 和 inference artifact 能够自描述 geometry backbone contract；
2. 为未来 PointNet / Transformer / Mesh GNN 一类主干接入预留统一入口；
3. 避免把当前 fixed branch-compatible 路线误解成长期唯一方案。

必须强调：

- 当前默认训练并没有切换到原生 variable-length geometry tokens；
- 当前 generic geometry 的训练与推理仍然主要通过兼容层映射到 fixed branch-compatible 表示；
- 当前也没有完成原生 geometry latent conditioning 的主训练闭环。

### 2.5 为什么当前重点仍是 2D

当前系统仍然是二维翼型代理主线。原因不是三维不重要，而是二维问题在当前阶段更适合作为框架验证平台：

- 数据可得性更稳定；
- 训练代价更低；
- 几何、采样、surface、slice 和 feature 输出更容易统一；
- 能先验证通用 schema、adapter、capability、branch input compatibility 是否成立。

因此，本轮结论应理解为“二维可扩展代理框架已经更稳”，而不是“系统已经具备三维工业替代能力”。

## 3. 本轮训练设置

### 3.1 数据设置

本轮仍以 AirfRANS heavy 作为默认训练与测试数据集。数据属性可概括为：

- 数据类型：二维、不可压缩、亚声速 RANS 翼型数据；
- 主监督场变量：`u, v, p, nut`；
- 主监督标量：`cl, cd`；
- 表面相关输出：`pressure_surface` 与 `cp_surface`；
- 分析型输出：slice 场采样、高梯度区域分析等。

本轮使用的主采样规模是：

- query points：1024
- surface points：256

这一设置比早期 smoke 级训练更接近正式实验口径，能够更好覆盖表面分布和局部场变化。

### 3.2 模型设置

本轮仍采用 DeepONet：

- hidden dimension：256
- latent dimension：256
- branch layers：4
- trunk layers：4
- Fourier features：64
- scalar head hidden dimension：128
- feature head output dimension：2

这里新增启用的 `feature head` 不是为了伪造新的真实监督，而是为了在现有 AirfRANS 可支持的分析输出基础上，增强 pressure-gradient/high-gradient 一类分析型结果。

### 3.3 训练策略

本轮训练参数如下：

- epoch：20
- batch size：4
- optimizer：Adam
- learning rate：1e-3
- weight decay：1e-6
- scheduler：CosineAnnealing
- mixed precision：关闭
- grad clip norm：1.0
- early stopping patience：8

### 3.4 Loss 组成

本轮并不是只训练基础场回归，而是采用了分析增强版多任务损失：

- field loss：监督 `u, v, p, nut`
- scalar loss：监督 `cl, cd`
- surface loss：监督 `cp_surface`
- slice loss：启用
- feature loss：启用
- physics loss：启用
- boundary loss：启用

权重大致为：

- field：1.0
- surface：0.5
- scalar：0.5
- slice：0.05
- feature：0.02
- physics：0.05
- boundary：0.05

这意味着本轮训练既要保证基本流场和气动系数不崩，又要尝试把分析型输出一起拉进训练闭环，因此训练难度高于早期只做基础输出的版本。

## 4. 本轮完整流程

本次完整跑通了如下流程：

1. 使用 AirfRANS 默认主路径进行正式重训练；
2. 使用最佳 checkpoint 进行 test split 评测；
3. 导出 test 集评测报告和可视化图像；
4. 额外使用通用二维翼型表面点输入进行一次推理演示；
5. 导出 analysis bundle，包括 surface、slice、feature 和标量结果。

因此，本轮不是只完成了“训练一个模型”，而是完成了“训练、评测、推理、导出、可视化”全链路闭环验证。

### 6.3 如何解读这组对比

这组对比非常重要，因为它揭示了当前系统所处的真实阶段：

**第一，本轮结果显著优于 smoke run。**

这说明当前重构后的训练主线是有效的，不是“只能跑通但学不好”。尤其：

- `cl / cd` 已经恢复到接近较优历史结果的量级；
- `field`、`slice` 和 `cp_surface` 都明显优于 3 epoch 兼容 run；
- 这证明 branch input 兼容式改造没有破坏默认 AirfRANS 主训练链路。

**第二，本轮还没有追上历史最佳分析增强结果。**

主要差距体现在：

- field RMSE 偏高；
- `cp_surface` 还不够低；
- feature 类指标明显弱；
- 尤其 `pressure_gradient_indicator` 几乎失效。

这说明当前“架构更稳”不等于“所有分析型输出都已经达到最好状态”。  
换句话说，本轮更像是“兼容重构后的正式恢复性重训练”，不是“单纯追精度极限的一轮长训调参”。

## 9. 当前效果的总体评价

本轮结果可以用三句话概括：

1. **工程上是成功的。**  
   默认 AirfRANS 主训练链路没有被兼容式重构破坏，完整流程已经跑通。

2. **主任务结果是可用的。**  
   `u/v/p/nut`、`cl/cd`、`cp_surface`、slice 等输出已经能够形成稳定的训练和评测闭环。

3. **分析型 feature 仍然偏弱。**  
   `high_gradient` 和 `pressure_gradient_indicator` 目前还不够强，说明 analysis 分支的训练设计还需要继续优化。

如果把当前效果分层评价：

- 对“主训练链路是否稳定”这个问题，答案是：稳定。
- 对“当前结果是否已经达到最好版本”这个问题，答案是：还没有。
- 对“当前结果是否已经足够支撑继续往通用 CFD surrogate 框架推进”这个问题，答案是：足够。

## 10. 当前仍然存在的主要问题

### 10.1 场误差与表面误差仍有下降空间

虽然本轮已经明显优于 smoke run，但 `field_rmse` 和 `cp_surface_rmse` 仍高于历史较优结果。  
这意味着当前版本在“稳定性”上已经恢复，但在“精度上限”上还没有完全找回之前最好状态。

### 10.2 Feature 分支仍然是短板

`pressure_gradient_indicator_f1 = 0` 是最值得警惕的指标。  
这说明当前 feature 分支虽然接口齐全、训练也打开了，但对显式梯度类伪标签的学习质量仍然不够。

### 10.3 physics loss 仍然更像正则，而不是直接提升主指标的核心来源

从训练过程可以看出，physics loss 在波动，但主指标下降主要还是来自 field/surface/scalar 路线。  
这说明当前物理约束设计是有价值的，但它还没有形成“强驱动精度提升”的状态。

### 10.4 generic geometry 的数值泛化还不能过度承诺

接口已经成立，兼容推理已经成立，但这不等于系统已经对任意通用二维翼型具备可靠精度。  
这部分在当前阶段只能被表述为“兼容能力已打通”，不能被包装成“泛化问题已解决”。

### 10.5 future native geometry backbone 仍只是接口预留

阶段 7 之后，仓库已经具备了 future native geometry backbone 的 contract 和实验性接口骨架，但这不代表：

- 已经完成原生 PointNet / Transformer / Mesh GNN 主干接入；
- 已经完成 variable-length geometry token 的默认训练路径；
- 已经完成面向三维 geometry/operator 的升级。

当前只能被准确表述为：

- 已完成接口级预留；
- 已完成 checkpoint / inference metadata 自描述；
- 尚未进入默认训练主路径。

## 11. 后续改进计划

### 11.1 继续优化 surface 路线

下一步最值得投入的是继续压低 `cp_surface_rmse`，因为：

- `cp_surface` 是气动解释中最重要的中间结果之一；
- 它又直接影响 surface analysis、设计筛选和汇报解释质量；
- 当前它已经可用，但仍未回到历史最佳水平。

优先方向包括：

- 调整 surface loss 权重；
- 重新平衡 field/surface/scalar/slice/feature 的多任务权重；
- 针对表面区域做更有针对性的采样强化。

### 11.2 重新设计 feature 伪标签与类别平衡

当前 feature 指标弱，不太像容量问题，更像伪标签质量和类别不平衡问题。  
后续应重点做：

- 重新定义 pressure-gradient 伪标签；
- 调整正负样本比例；
- 评估是否需要把部分 feature 先降级为纯 derived output。

### 11.3 比较 legacy 与 encoded geometry 两条 branch 输入通道

当前默认训练仍走 `legacy_fixed_features`。  
后续最自然的一步，是在同样的数据和训练预算下，直接比较：

- `legacy_fixed_features`
- `encoded_geometry`

这样才能判断新几何兼容通道到底只是“能跑”，还是已经开始具备实际收益。

### 11.4 在保持二维主线稳定的前提下继续推进通用框架

当前统一 sample schema、adapter、capability 和 branch input compatibility 已经打通。  
后续更合适的路径不是急着跳三维，而是：

1. 继续引入更多二维数据适配器；
2. 让 trainer/evaluator 对 capability 的依赖更彻底；
3. 把 unified schema 逐步从桥接层推进成真正内部主表示。

### 11.5 native geometry backbone 升级的前提条件

如果未来真的要把系统从 fixed branch-compatible 几何条件化，推进到原生可变长 geometry backbone，建议先满足这些前提：

1. dataset/collate 能稳定支持 variable-length geometry batching；
2. 明确 geometry latent / geometry tokens 与 query decoder 的 conditioning 契约；
3. inference / checkpoint / export 已能稳定描述 geometry backbone 类型与激活状态；
4. generic 2D dataset 的覆盖足够支撑这类主干训练；
5. 在二维 generic geometry 上先完成受控对比，再考虑更重的主干升级。

### 11.6 为未来 3D 做接口级准备，而不是直接开 3D 训练

真正的三维扩展不应建立在“二维还不够稳”的前提上。  
当前更合理的策略是：

- 保持二维主线继续优化；
- 让 geometry/query/head/schema 的接口不再写死二维；
- 等二维通用化真正稳定后，再进入三维 surface surrogate 阶段。

## 12. 结论

本轮重训练最重要的结论不是“又得到了一组新数字”，而是：

**在完成兼容式架构升级之后，当前系统已经能够继续以 AirfRANS 作为默认主数据集，稳定完成训练、评测、推理与 analysis bundle 导出，并保持主要输出能力不回退。**

从结果上看：

- 主任务已经恢复到可用区间；
- 与短训练兼容 run 相比，性能显著改善；
- 与历史最佳版本相比，仍有差距；
- feature 路线仍是短板；
- 通用二维翼型输入已经可演示，但还不能过度宣称为成熟泛化能力。

因此，当前系统最准确的定位是：

**它已经是一个可运行、可评测、可导出、具备通用化雏形的二维 CFD surrogate 框架，而不是只会 AirfRANS 的一次性实验脚本；但它距离“高精度、强泛化、可直接外推到三维”的目标还有明确的优化空间。**

补充一句非常重要的边界说明：

**到阶段 7 为止，仓库已经为 future native variable-length geometry backbone 做了清晰接口预留，但默认训练主线仍然是当前稳定的 fixed branch-compatible path。不要把这一步误解成已经完成原生几何主干升级。**
